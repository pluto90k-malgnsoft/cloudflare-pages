# v2.1 Aresample(async=1) 적용 여부 감지 엔진

FFmpeg `-af aresample=async=1` 옵션의 적용 여부를 자동 감지하여 안드로이드 크롬 재생 불가 원인을 진단합니다.

## 배경

- **문제**: 특정 HLS 영상이 안드로이드 크롬에서 재생되지 않음
- **원인**: FFmpeg 인코딩 시 `-af aresample=async=1` 옵션 미적용
- **효과**: 이 옵션은 오디오 타임스탬프의 연속성을 강제 보장
  - 갭 발생 시 → 무음(silence) 삽입
  - 겹침 발생 시 → 오디오 트리밍
  - 결과: 오디오 PTS가 항상 단조 증가하고 빈틈없이 연결

## 변경 사항

### [js/base.js] - 세그먼트 내 오디오 통계 계산
- `parseSegmentTimestamps()`에서 반환하는 오디오 객체에 3개 필드 추가:
  - `actualDuration`: 세그먼트 내 오디오 실제 재생 시간 (lastPTS - firstPTS)
  - `avgInterval`: 오디오 PES 패킷 간 평균 PTS 간격
  - `intervalStdDev`: PTS 간격의 표준편차

### [js/base.py] - 진단 엔진 v2.1

#### 3대 분석 지표
| 지표 | 설명 | 정상 기준 | 이상 기준 |
|:---|:---|:---|:---|
| **Audio Continuity Gap** | 세그먼트 경계 오디오 연속성 | <150ms (~AAC 6프레임) | >150ms 또는 대형 음수 |
| **Duration Mismatch** | 실제 오디오 duration vs #EXTINF 선언값 | <100ms | >100ms |
| **Audio PTS StdDev** | 세그먼트 내 오디오 샘플 간격 균일성 | <10ms | >10ms |

#### 판정 로직
- **GAP/타임라인 역행 세그먼트는 분석에서 제외** (소스 결함과 Aresample 미적용을 분리 판정)
- 정상 연속 세그먼트에서 3개 지표 중 **하나라도** 임계값 초과 → `aresample=async=1 미적용 의심`
- 판정 우선순위: `타임라인 역행` > `진동` > `거대 GAP` > **`Aresample 미적용`** > `복구됨` > `오차`

#### UI 변경
- **대시보드**: `Audio 연속성 Gap`, `Duration 편차`, `Aresample 판정` 카드 3개 추가
- **분석 테이블**: `A-Gap` 컬럼 추가, `AUDIO-GAP`(보라) / `DUR-DRIFT`(주황) 배지 추가
- **진단 보고서**: `Aresample(async=1) 분석 보고서` 전용 섹션 추가
  - 미적용 시: 근거 증거 목록 + 권장 조치 안내
  - 적용 시: 정상 확인 + 수치 요약

## 검증 방법

### 테스트 시나리오
1. **aresample=async=1 적용 영상**: Audio Gap이 ~21ms로 안정적, Duration 편차 <30ms → "적용됨" 판정
2. **aresample=async=1 미적용 영상**: Audio Gap이 불규칙(>50ms), Duration 편차 >30ms → "미적용 의심" 판정
3. **두 영상의 분석 결과 비교**로 옵션 유무에 따른 차이 확인

### FFmpeg 인코딩 예시
```bash
# aresample 적용 (안드로이드 크롬 재생 가능)
ffmpeg -i input.mp4 -af aresample=async=1 -hls_time 6 -hls_list_size 0 output.m3u8

# aresample 미적용 (안드로이드 크롬 재생 불가 가능성)
ffmpeg -i input.mp4 -hls_time 6 -hls_list_size 0 output.m3u8
```

## 실측 데이터 기반 수정 이력

### v2.1.1 — Aresample 오탐 수정
**테스트 소스**: `V22133.m3u8` (aresample=async=1 적용, 안드로이드 크롬 정상 재생)

**발견된 문제**:
1. GAP/타임라인 역행 세그먼트(#4, #5, #9)의 A-Gap(-18000ms, -21000ms)이 평균을 오염 → 6454ms로 왜곡
2. Audio Gap 50ms 기준 과민 → AAC 프레임 배수(69.7ms, 92.9ms)를 이상으로 오판
3. Duration 30ms 기준 과민 → PES 경계 정렬 편차(30~50ms)를 이상으로 오판
4. StdDev 2ms → 5ms 기준에서도 6.787ms로 오탐 지속

**수정**:
- GAP/역행 세그먼트를 aresample 지표 수집에서 제외
- 임계값 조정: Audio Gap 50→150ms, Duration 30→100ms, StdDev 2→5ms
- GAP 세그먼트에서 AUDIO-GAP/DUR-DRIFT 배지 억제

### v2.2 — 콘텐츠 경계(Content Boundary) 감지 도입
**발견된 문제**:
1. `#EXT-X-DISCONTINUITY` 태그가 없지만 PTS 역행 존재 → "FATAL: TIMELINE REVERSAL" 오판
2. 실제로는 VOD 편집/연결 지점으로, 안드로이드 크롬에서 정상 재생됨
3. StdDev 5ms 기준에서 여전히 6.787ms로 aresample 오탐

**핵심 발견**: PTS 역행은 콘텐츠 구조적 특성이며, `#EXT-X-DISCONTINUITY` 태그 유무와 무관하게 정상 재생 가능

**수정**:
- PTS 역행이 동반된 GAP을 **콘텐츠 경계**(content boundary)로 분류
- 콘텐츠 경계를 타임라인 역행 FATAL 판정에서 제외
- 연속 구간 내 역행만 진짜 FATAL로 판정
- `DISC` 배지(파란색, #0891b2) 및 파란 배경 행 스타일 추가
- "정상 (Discontinuity N개)" 판정 로직 추가
- StdDev 임계값 5→10ms (PES 파싱 특성 반영)
